# Локальный запуск проекта (Django backend)

Пошаговая инструкция для разработки на своей машине. Команды выполняются из **корня репозитория** (каталог, где лежит `manage.py`).

## Что нужно установить

- **Python** 3.10+ (рекомендуется та же мажорная версия, что и на сервере)
- **PostgreSQL** (локально или Docker)
- **Git**

Опционально: на macOS удобно ставить PostgreSQL через Homebrew (`brew install postgresql@14` и т.п.).

---

## 1. Клонирование и переход в проект

```bash
git clone <url-репозитория>
cd danadjango
```

(Если репозиторий уже склонирован — достаточно `cd` в каталог с `manage.py`.)

---

## 2. Виртуальное окружение Python

Создать окружение (один раз):

```bash
python3 -m venv venv-danadjango
```

Активировать:

```bash
# macOS / Linux
source venv-danadjango/bin/activate

# Windows (cmd)
venv-danadjango\Scripts\activate.bat

# Windows (PowerShell)
venv-danadjango\Scripts\Activate.ps1
```

Дальше в инструкции предполагается активированное окружение.

---

## 3. Зависимости

```bash
pip install --upgrade pip
pip install -r requirements.txt
```

Первый запуск EasyOCR может скачать модели — это нормально и может занять время.

---

## 4. PostgreSQL

Убедитесь, что сервер PostgreSQL запущен.

**Пример (macOS, Homebrew):**

```bash
brew services list
brew services start postgresql@14
```

Создайте базу данных. Имя по умолчанию в настройках — `meta-university_db`.

Войдите в консоль `psql` под пользователем с правами создания БД (часто это ваш системный пользователь или `postgres`):

```bash
psql postgres
```

В консоли SQL:

```sql
CREATE DATABASE meta-university_db;
```

Пользователь и пароль для подключения Django должны совпадать с тем, что вы укажете в `.env` (см. ниже). Если вы используете стандартного пользователя `postgres` с паролем — задайте пароль через `ALTER USER` в PostgreSQL или создайте отдельного пользователя и выдайте ему права на эту БД.

---

## 5. Файл `.env`

В **корне проекта** (рядом с `manage.py`) создайте файл `.env`, если его ещё нет. Пример минимальной конфигурации для локальной разработки:

```env
SECRET_KEY=сгенерируйте-свой-секретный-ключ-для-разработки
DEBUG=True
ALLOWED_HOSTS=localhost,127.0.0.1

DB_NAME=meta-university_db
DB_USER=postgres
DB_PASSWORD=ваш_пароль
DB_HOST=localhost
DB_PORT=5432
```

- `SECRET_KEY`: для продакшена обязательно уникальный секрет; для локальной машины достаточно длинной случайной строки.
- Параметры `DB_*` должны соответствовать реальной учётной записи PostgreSQL.

Переменные читаются через `python-decouple` из `danadjango/settings.py`.

---

## 6. Миграции

```bash
python manage.py migrate
```

Команда `makemigrations` нужна, когда вы меняете модели; для чистого клона из репозитория обычно достаточно только `migrate`.

---

## 7. Начальные данные

```bash
python manage.py init_ocr
python manage.py init_data
```

После `init_data` (если команда создаёт демо-пользователей — см. вывод в терминале и корневой `README.md` проекта) обычно доступны, например:

- администратор: `admin` / `admin123`
- тестовый пользователь: `testuser` / `test123`

Смените пароли, если поднимаете что-то доступное из сети.

---

## 8. Статика (по желанию для локалки)

```bash
python manage.py collectstatic --noinput
```

Для `runserver` с `DEBUG=True` статика из `STATICFILES_DIRS` часто отдаётся и без полного `collectstatic`; для проверки «как в проде» сборка полезна.

---

## 9. Запуск сервера разработки

```bash
python manage.py runserver
```

По умолчанию: **http://127.0.0.1:8000/**

Полезные адреса:

| Назначание        | URL |
|-------------------|-----|
| Админка           | http://127.0.0.1:8000/admin/ |
| Swagger           | http://127.0.0.1:8000/swagger/ |
| ReDoc             | http://127.0.0.1:8000/redoc/ |

Список основных API и примеры запросов — в корневом **`README.md`**.

---

## 10. Чат и OCR с OpenAI / внешними API

- Эндпоинты чата: префикс `/api/ai/...` (нужен JWT).
- Ключ OpenAI и настройки — по документации: **`документация/AI_CHAT_BACKEND.md`**, **`документация/CHATGPT_INTEGRATION.md`**, для клиента Flutter — **`документация/FLUTTER_AI_CHAT.md`**.

Без настроенного ключа в админке/настройках соответствующие функции могут возвращать ошибки — это ожидаемо.

---

## Типичные проблемы

- **`could not connect to server`** — PostgreSQL не запущен или неверные `DB_HOST` / `DB_PORT`.
- **`password authentication failed`** — неверный `DB_USER` / `DB_PASSWORD` в `.env`.
- **`database "meta-university_db" does not exist`** — создайте БД (раздел 4).
- **Порт 8000 занят** — `python manage.py runserver 8001`
